# What we're doing
ObsidiClaw is a **self-improving memory and context injection system** for an AI agent ("Pi"). It gives Pi a knowledge graph of tools and insights, logs its runs, compares outcomes, and writes durable lessons back to the graph. The goal: an agent that gets smarter each session through structured memory.

Full spec: see `md_db/index.md` and `.claude/Conceptual_Plan.md`.

# Key Design Decisions
- `md_db/tools/` nodes point to **live tools** that fetch real-time facts — never hardcode facts
- `md_db/concepts/` stores **durable agent insight** (failure modes, best practices, heuristics)
- `md_db/code/` stores **code mirror notes** — generated by mirror scripts, enriched by summarizer
- Use **Conda** for all Python commands
- **MCP boundary**: `ContextEngine` is only accessible via `knowledge/engine/mcp/mcp-server.ts` — Pi never holds a direct engine reference
- **Shared infrastructure**: `createObsidiClawStack()` in `entry/stack.ts` creates ContextEngine, RunLogger, NoteMetricsLogger, WorkspaceRegistry. Both `entry/extension.ts` (Pi TUI) and `agents/orchestrator/run.ts` (headless) consume it
- **Two entry points**: `pi` (interactive, full stack) and `npx tsx agents/orchestrator/run.ts` (headless/scripting/gateway)
- **Python owns retrieval**: `knowledge/graph/` runs as subprocess. VectorStoreIndex for vectors + SimplePropertyGraphStore for wikilink graph. TS bridge via JSON-RPC stdin/stdout. `conda run` doesn't forward stdin on Windows — spawn conda env Python directly
- **Incremental reindex**: not a scheduled job. Chokidar watcher in `entry/stack.ts` watches `md_db/` and calls `engine.incrementalUpdate()` (1.5s debounce). Live only when Pi is running; full reindex on next startup otherwise
- **Workspace registry**: persistent JSON at `.obsidi-claw/workspaces.json`. Agent registers/unregisters codebases via MCP tools at runtime. Stack auto-registers itself on first boot. Each workspace gets `md_db/{mode}/{name}/` subfolder. One unified graph with `workspace` metadata filtering in LlamaIndex
- **Two workspace modes**: `"code"` runs the mirror pipeline + watcher; `"know"` is a future conversational knowledge mode (directory created, no pipeline yet)
- **Code mirror pipeline**: source → `mirror-codebase.ts/py` → `md_db/code/{workspace}/*.md` → chokidar mirror watcher spawns `summarize-worker.ts` (child process) to add `## Summary` + tags → reindex watcher reindexes live. `setup-tasks.ts` registers `force-summarize.ts` as an OS schtask for offline catch-up
- **Staleness is mtime-only**: `source mtime > mirror mtime` → summarizer runs. No tracking files — filesystem is the state
- **Automation boundary**: in-runtime automation (watchers, event handlers) lives in `automation/jobs/`; out-of-runtime automation (mirror, summarize) lives in `automation/scripts/` as standalone scripts. Pi never manages OS tasks. `setup-tasks.ts` is a one-time CLI tool that registers schtasks entries pointing at standalone scripts
- **Debug events**: `ContextEngine` emits `ce_*` events at every state transition. All events → `trace` table via `RunLogger.logEvent()`. No JSONL — runs.db is the single event sink
- **Startup injection**: `before_agent_start` injects `preferences.md` only — no RAG on startup. Pi uses `retrieve_context` for on-demand lookup
- **Index notes filtered**: `type === "index"` notes excluded from retrieval and graph expansion
- **Session/Trace schema**: `sessions` table (multi-prompt container) + `trace` table (17 structured columns: source/target/action/status/phase/data/summary/error + monotonic seq per session). `runs` table is planned but not yet built — `run_id` in trace is a loose correlation string only
- **Split databases**: `runs.db` owns operational event log (sessions, runs, trace). `notes.db` owns note retrieval analytics (note_hits, synthesis_metrics, context_ratings, note_lifecycle, prune clusters). Schema is 3-tier-aware from the start (tier, note_type, symbol_kind, edge_label columns)
- **Trace modules**: canonical names in `logger/trace-modules.ts`: `orchestrator`, `pi_session`, `context_engine`, `extension`, `insight_engine`, `logger`, `user`, `tool:<name>` (dynamic)
- **LLM config**: `OBSIDI_LLM_HOST` env var (must be set in `.env`). Defaults to `localhost:11434` which is wrong for this setup — always set explicitly
- **No console.log/warn/error**: Pi TUI renders in a styled terminal — raw stdout pollutes the UI. Background/worker errors are swallowed silently, returned as null, or logged to SQLite. If something truly needs user attention it surfaces through Pi's UI layer, not stdout.

- Hybrid RAG: Python LlamaIndex vector seeds + graph BFS expansion + tag boosting
- Context synthesis: LLM rewrites raw notes into query-focused context (requires `OBSIDI_LLM_HOST` in `.env`)
- Code mirror pipeline: TS + Python source → `md_db/code/` mirrors → summarizer adds summaries + tags
- Incremental reindex: chokidar watcher calls `engine.incrementalUpdate()` on live engine when md_db changes
- MCP boundary: context engine only through MCP server
- Comprehensive event logging: unified SQLite trace table. RunEvent union is Pi TUI only — no dead orchestrator variants. `agent_run_start` (once per prompt) + `agent_turn_end` (per LLM turn) + `prompt_complete` (with durationMs) all emitted correctly
- Split databases: runs.db (operational log) + notes.db (retrieval analytics, 3-tier-aware schema, prune clusters, note lifecycle)
- **Clean automation boundary**: `JobScheduler` removed from Pi runtime. Out-of-runtime scripts in `automation/scripts/`. `setup-tasks.ts` for one-time schtasks registration
- Session/trace schema: `sessions` table + clean `trace` table (17 columns, no payload blob). `run_id` is a loose per-prompt correlation string in trace; no backing `runs` table yet
- Three-tier code mirror notes: tier-1 (codeSymbol), tier-2 (codeUnit), tier-3 (codeModule) — generated by `mirror-codebase.ts --force`. Tier-3 notes named `{dirName}_module.md` alongside their directory; eliminates collision with same-named tier-2 notes (e.g. `orchestrator.ts` inside `orchestrator/`)
- Typed graph edges: DEFINED_IN, BELONGS_TO, CONTAINS, CONTAINS_SYMBOL, CALLS, IMPORTS, LINKS_TO — all verified in indexer
- Tier-aware retrieval: going-up always (0.8×/0.5×), going-down selective (≥0.6 seed threshold), sideways on query overlap (0.6×)
- **Force-mirror:** `npx tsx automation/scripts/mirror-codebase.ts --force` (also `mirror-codebase-py.ts --force`) — reads `workspaces.json`, runs for all active workspaces with correct workspace prefix
- **Workspace registry**: persistent JSON at `.obsidi-claw/workspaces.json`. MCP tools: `register_workspace`, `list_workspaces`, `unregister_workspace`. Auto-registers self on first boot. `retrieve_context` accepts optional `workspace` param for scoped search
- **Multi-codebase support**: each workspace gets its own `md_db/code/{name}/` subfolder with per-workspace wikilink prefixes. One unified graph with metadata-based workspace filtering
- **retrieve_context budget raised**: 3000 → 10000 chars (~750 → ~2500 tokens) — prevents synthesis output truncation
- **find_path MCP tool**: graph path retrieval between two fuzzy endpoints. BFS (bidirectional, records edge direction) over SimplePropertyGraphStore. Endpoint resolution: exact → partial path → vector → keyword fallback. `formatPathContext()` renders step-by-step with ↓ EDGE_LABEL annotations. Passes through existing ContextReviewer. Works in degraded mode (BFS needs no embeddings)
- **Startup re-embed bug fixed**: `compute_md_db_hash()` uses content-based hashing (MD5 of file bytes) instead of mtime-based. Immune to linter frontmatter rewrites, field reordering, and any non-semantic file touches
- **Python server single-threaded**: removed ThreadPoolExecutor + `_engine_lock` + `_stdout_lock` that caused lock inversion deadlocks (progress_cb → send_notification while holding engine lock). Handlers run synchronously; stdin buffers naturally queue concurrent requests
